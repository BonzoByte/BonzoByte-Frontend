What this model view shows

This tab reveals the structured feature groups behind the neural-network-based match probability shown in the modal. It is designed as a transparency layer, exposing the main input families that contribute to the final prediction instead of presenting the result as a black box.

Rather than relying on a single heuristic or a bookmaker-style shortcut, the displayed probability comes from a curated modelling pipeline built around explicit feature selection, grouped signals, and controlled promotion into the production prediction field.

How to read this tab

Each sub-section represents one feature family used by the model. Some groups describe the tournament and match context, others reflect core win-probability signals, rating-based strength estimates, normalized transformations, derived precision-style signals, and optional head-to-head inputs.

The purpose of this tab is interpretability. It helps advanced users understand that the final probability is generated from a structured analytical system, not from a single opaque score.

Model notes

The current production setup follows a trusted feature policy, treats ATP and WTA as separate modelling domains, and retains head-to-head information only when it improves the final model profile. The feature groups shown here are a compact view of the promoted model inputs behind the stored match probability.

What this section represents

The Context section captures the match environment around the players rather than the players themselves. It includes tournament structure, playing surface, event strength, round context, and a compact set of competition-level signals that help the model understand where and under what conditions the match takes place.

These inputs provide situational framing for the prediction. They do not attempt to describe player strength directly, but instead supply the surrounding context that can influence how rating-based and probability-based signals should be read.

How to read these values

Some fields are categorical identifiers such as surface or tournament type, while others are normalized or transformed quantities that represent event strength, surface profile, experience balance, or structural match conditions. Together, they form the environmental layer of the model input.

What this section represents

WP Core contains the central probability-style signals that sit closest to the model’s base match expectation. These values summarize different win-probability perspectives and related rating-informed projections that give the model an immediate sense of which side appears stronger before deeper feature interaction.

This is one of the most direct signal families in the NN view. While it is still only one part of the full model input, it represents the core probabilistic layer on top of which contextual and transformed signals are interpreted.

How to read these values

Most values in this section are probability-like inputs derived from structured pre-model calculations. They should be understood as calibrated directional signals, not as standalone final predictions. The production probability is generated only after these values are combined with the wider feature set.

What this section represents

Raw TrueSkill contains the direct strength estimates and uncertainty measures used as rating-based inputs for both players. These values describe underlying player quality from the rating system’s perspective before normalization or additional transformations are applied.

In practical terms, this section reflects how the model sees player strength and confidence at the rating layer. It provides one of the main structural foundations for the downstream comparison features used elsewhere in the tab.

How to read these values

Mean values represent estimated player strength, while SD values represent the remaining uncertainty around that estimate. Because the model consumes multiple rating variants, these inputs are shown as a grouped family rather than as a single score per player.

What this section represents

Normalized TrueSkill contains transformed rating-based features that make raw signals easier for the model to compare across matches, players, and competitive contexts. These values help stabilize scale differences and improve consistency when the network processes the broader feature set.

This section is less about direct human interpretation and more about preparing rating information in a model-friendly form. It acts as a bridge between raw strength estimates and higher-level derived comparisons.

How to read these values

Values here are typically normalized or transformed versions of rating-based comparisons. They are intended to improve comparability and numerical stability, not to function as standalone headline metrics.

What this section represents

Derived TrueSkill contains compact comparison features built from the broader rating and probability foundation. These values capture higher-level relationships such as precision-style or weighted comparative signals that can help the model distinguish between superficially similar matches.

This is a smaller feature family, but it is useful because it compresses more complex rating behaviour into a concise set of structured inputs that the model can consume efficiently.

How to read these values

The features shown here are not raw measurements. They are engineered comparison signals derived from upstream rating and probability components, designed to express match-level separation in a more model-friendly way.

What this section represents

The Head-to-Head section contains historical interaction signals between the two players when that information is included by the promoted model version. These values are treated as a secondary input family rather than as the primary source of the final probability.

In the current production setup, head-to-head information is retained only where it improves the final model profile. It acts as supporting evidence on top of the broader contextual, probabilistic, and rating-based layers.

How to read these values

These inputs summarize prior interaction patterns in a compact way. They can add useful context in some matchups, but they should be read as a complementary signal, not as a substitute for the rest of the model structure.
